# Task: Add Offline-Capable VS Code Extension Prefetch and Install Support

## Goal

Add a VS Code extension artifact workflow to `devcontainer-blueprints`.

The workflow should:

1. Take a list of VS Code extension IDs.
2. Resolve the **latest compatible version** of each extension for a target VS Code version.
3. Prefer the VSIX that matches the target container platform, such as `linux-x64` or `linux-arm64`.
4. Download all required VSIX files during an online prefetch phase.
5. Expand extension packs and extension dependencies recursively.
6. Produce a lockfile with exact extension IDs, versions, target platform, SHA256 hashes, and install order.
7. Install extensions during a Docker build with `--network=none`.
8. Install extensions into the remote VS Code Server extension directory for the intended remote user, usually `vscode`.

This should build on the earlier VS Code Server preinstall work.

The image build must not download extensions from the internet.

---

# Important Concepts

VS Code extensions are not selected by VS Code commit SHA directly.

They are selected by:

```text
target VS Code product version
extension package.json -> engines.vscode
target platform
extension kind
extension dependencies
extension packs
```

Example:

```text
VS Code product version: 1.124.2
VS Code commit:          40-character commit SHA
container platform:      linux-x64
server platform:         server-linux-x64
extension target:        linux-x64 or universal fallback
```

The VS Code Server must still match the VS Code client commit exactly.

Extensions are compatible with a VS Code version when:

```json
{
  "engines": {
    "vscode": "^1.90.0"
  }
}
```

matches the target VS Code product version.

---

# Important Caveat About UI Extensions

Some extensions are **UI extensions**. They run on the local VS Code client machine, not inside the remote container.

Some extensions are **workspace extensions**. They run inside the devcontainer.

The resolver must inspect:

```json
{
  "extensionKind": ["ui"]
}
```

or:

```json
{
  "extensionKind": ["workspace"]
}
```

Rules for this first implementation:

```text
extensionKind includes "workspace"
  -> container-installable

extensionKind is missing
  -> treat as container-installable unless proven otherwise

extensionKind is exactly ["ui"]
  -> classify as host-only and do not install in the container by default

extensionKind includes both "ui" and "workspace"
  -> container-installable, but record both kinds in the lockfile
```

The extension:

```text
ms-vscode-remote.vscode-remote-extensionpack
```

is likely host/client-side in purpose. The resolver should prefetch it if listed, but it should not force it into the container if its manifest says it is UI-only.

# Script 2: `scripts/prefetch-vscode-extensions.mjs`

Implement this as a robust Node script.

## Required CLI

Support:

```bash
./scripts/prefetch-vscode-extensions.sh
```

```bash
./scripts/prefetch-vscode-extensions.sh \
  --vscode-version 1.124.2 \
  --target-platform linux-x64
```

```bash
./scripts/prefetch-vscode-extensions.sh \
  --server-metadata artifacts/vscode-server/current-stable-server-linux-x64.json \
  --extensions-file config/vscode-extensions.txt
```

```bash
./scripts/prefetch-vscode-extensions.sh \
  --include-ui
```

```bash
./scripts/prefetch-vscode-extensions.sh \
  --include-prerelease
```

## Required Behavior

The script must:

1. Load defaults from `config/vscode-extensions.env`.
2. Load extension IDs from `config/vscode-extensions.txt`.
3. Resolve target VS Code product version from one of:
   - `--vscode-version`
   - `VSCODE_SERVER_METADATA_JSON.productVersion`
4. Resolve VS Code commit from:
   - `VSCODE_SERVER_METADATA_JSON.commit`
   - optional `--vscode-commit`
5. Query the Visual Studio Marketplace for each extension.
6. Prefer target-platform-specific VSIXs.
7. Fall back to universal VSIXs.
8. Download candidate VSIX files.
9. Extract `extension/package.json` from each VSIX.
10. Check `engines.vscode` against the target VS Code product version.
11. Check `extensionKind`.
12. Recursively resolve:
    - `extensionDependencies`
    - `extensionPack`
13. Compute SHA256 for each VSIX.
14. Write exact metadata and install order to a lockfile.
15. Be idempotent: if a matching VSIX already exists and SHA256 matches the lockfile, do not redownload it.
16. Fail clearly if no compatible version is found.

## Marketplace Query Details

Use:

```text
POST https://marketplace.visualstudio.com/_apis/public/gallery/extensionquery
```

Headers:

```text
Accept: application/json;api-version=3.0-preview.1
Content-Type: application/json
User-Agent: VSCode Offline Extension Prefetcher
```

Use query criteria like:

```json
{
  "filters": [
    {
      "criteria": [
        {
          "filterType": 7,
          "value": "publisher.extension"
        },
        {
          "filterType": 8,
          "value": "Microsoft.VisualStudio.Code"
        },
        {
          "filterType": 23,
          "value": "linux-x64"
        }
      ],
      "pageNumber": 1,
      "pageSize": 100,
      "sortBy": 0,
      "sortOrder": 0
    }
  ],
  "assetTypes": [
    "Microsoft.VisualStudio.Services.VSIXPackage"
  ],
  "flags": 243
}
```

Where:

```text
filterType 7  = Name
filterType 8  = InstallationTarget
filterType 23 = TargetPlatform
```

Use flags:

```text
IncludeVersions             = 1
IncludeFiles                = 2
IncludeVersionProperties    = 16
ExcludeNonValidated         = 32
IncludeInstallationTargets  = 64
IncludeAssetUri             = 128

Total = 243
```

If the target-platform query does not return a usable VSIX, retry without `TargetPlatform`.

## VSIX Download URL Selection

For each selected version, prefer the API-provided file URL:

```js
version.files.find(f => f.assetType === "Microsoft.VisualStudio.Services.VSIXPackage").source
```

If no usable file source exists, construct fallback URLs.

Primary fallback:

```text
https://marketplace.visualstudio.com/_apis/public/gallery/publishers/{publisher}/vsextensions/{extension}/{version}/vspackage
```

Secondary fallback:

```text
https://{publisher}.gallery.vsassets.io/_apis/public/gallery/publisher/{publisher}/extension/{extension}/{version}/assetbyname/Microsoft.VisualStudio.Services.VSIXPackage
```

For platform-specific extensions, the API-provided URL is preferred because it is more likely to point at the selected platform package.

## Candidate Selection Algorithm

For each extension ID:

1. Normalize to lowercase for comparisons.
2. Split `publisher.extensionName`.
3. Query Marketplace with `TargetPlatform = linux-x64` or `linux-arm64`.
4. Also query Marketplace without target platform for universal fallback.
5. Merge candidates.
6. Sort candidates newest-first by semver.
7. For each candidate:
   - skip prerelease versions unless `--include-prerelease` is set
   - download VSIX to a temporary path
   - extract `extension/package.json`
   - read `engines.vscode`
   - check `semver.satisfies(targetVscodeVersion, enginesRange, { includePrerelease: true })`
   - read `extensionKind`
   - classify as `container`, `host`, or `both`
   - accept newest compatible candidate
8. If no compatible candidate exists, fail.

## Extension Kind Classification

Use this logic:

```text
extensionKind missing:
  containerEligible = true
  hostOnly = false
  reason = "extensionKind missing; treated as workspace-capable"

extensionKind contains "workspace":
  containerEligible = true

extensionKind is exactly ["ui"]:
  containerEligible = false
  hostOnly = true

extensionKind contains both "ui" and "workspace":
  containerEligible = true
  hostOnly = false
```

When `INSTALL_CONTAINER_EXTENSIONS_ONLY=true`, do not put host-only extensions in the container install order.

Still include host-only extensions in the lockfile under:

```json
"hostOnlyExtensions": []
```

## Dependency and Extension Pack Handling

For each accepted VSIX, read from `extension/package.json`:

```json
{
  "extensionDependencies": [],
  "extensionPack": []
}
```

Rules:

```text
extensionDependencies:
  These are hard dependencies.
  Resolve and install them before the extension that declares them.

extensionPack:
  These are bundled recommendations/pack members.
  Resolve and install pack members before installing the pack extension itself.

extension pack extension itself:
  Install last, after all pack members.
```

Build a directed graph:

```text
dependency -> dependent
pack member -> extension pack
```

Then topologically sort the graph.

If a dependency is host-only but a container extension depends on it, record a warning and fail unless `--allow-host-only-dependency` is supplied.

# Script 3: `scripts/install-vscode-extensions.sh`

## Purpose

Install prefetched VSIX files during Docker build using the preinstalled VS Code Server CLI.

This script must not use the internet.

It must install in the order specified by the lockfile.

## Required CLI

Support:

```bash
install-vscode-extensions.sh \
  --lock /opt/vscode-extensions/vscode-extensions.lock.json \
  --user vscode
```

Support explicit commit:

```bash
install-vscode-extensions.sh \
  --lock /opt/vscode-extensions/vscode-extensions.lock.json \
  --user vscode \
  --commit <vscode-commit>
```

Support explicit server binary:

```bash
install-vscode-extensions.sh \
  --lock /opt/vscode-extensions/vscode-extensions.lock.json \
  --user vscode \
  --code-server /home/vscode/.vscode-server/cli/servers/Stable-<commit>/server/bin/code-server
```

## Required Behavior

The script must:

1. Read the lockfile with `jq`.
2. Determine target VS Code commit from:
   - `--commit`
   - `lockfile.targetVscodeCommit`
3. Find the VS Code Server CLI in this order:
   - explicit `--code-server`
   - `/home/<user>/.vscode-server/cli/servers/Stable-<commit>/server/bin/code-server`
   - `/home/<user>/.vscode-server/bin/<commit>/bin/code-server`
   - `code` on PATH, only as a fallback
4. Use extensions dir:

```text
/home/<user>/.vscode-server/extensions
```

5. Install only entries in:

```json
containerInstallOrder
```

6. Before installing each extension:
   - verify the VSIX file exists
   - verify SHA256 matches lockfile
7. Install each extension with:

```bash
code-server \
  --extensions-dir /home/vscode/.vscode-server/extensions \
  --user-data-dir /tmp/vscode-server-user-data \
  --install-extension /path/to/extension.vsix \
  --force
```

8. Run the install as the remote user, not root, when possible.
9. After installation, run:

```bash
code-server \
  --extensions-dir /home/vscode/.vscode-server/extensions \
  --list-extensions \
  --show-versions
```

10. Verify every container extension from the lockfile appears in the installed extension list.
11. `chown -R vscode:vscode /home/vscode/.vscode-server`.

## Implementation Requirements

Create a robust Bash script.

Skeleton:

```bash
#!/usr/bin/env bash
set -euo pipefail

usage() {
  cat <<'USAGE'
Usage:
  install-vscode-extensions.sh --lock LOCKFILE [options]

Options:
  --lock LOCKFILE        Required lockfile path.
  --user USER            Remote user. Default: vscode.
  --commit COMMIT        VS Code commit SHA. Defaults to lockfile targetVscodeCommit.
  --code-server PATH     Explicit code-server binary.
  --extensions-dir DIR   Explicit extensions dir.
  -h, --help             Show help.
USAGE
}

LOCKFILE=""
REMOTE_USER="vscode"
COMMIT=""
CODE_SERVER=""
EXTENSIONS_DIR=""

while [ "$#" -gt 0 ]; do
  case "$1" in
    --lock)
      LOCKFILE="$2"
      shift 2
      ;;
    --user)
      REMOTE_USER="$2"
      shift 2
      ;;
    --commit)
      COMMIT="$2"
      shift 2
      ;;
    --code-server)
      CODE_SERVER="$2"
      shift 2
      ;;
    --extensions-dir)
      EXTENSIONS_DIR="$2"
      shift 2
      ;;
    -h|--help)
      usage
      exit 0
      ;;
    *)
      echo "ERROR: unknown argument: $1" >&2
      usage
      exit 1
      ;;
  esac
done

if [ -z "${LOCKFILE}" ]; then
  echo "ERROR: --lock is required." >&2
  usage
  exit 1
fi

if [ ! -f "${LOCKFILE}" ]; then
  echo "ERROR: lockfile does not exist: ${LOCKFILE}" >&2
  exit 1
fi

for cmd in jq sha256sum; do
  if ! command -v "${cmd}" >/dev/null 2>&1; then
    echo "ERROR: required command not found: ${cmd}" >&2
    exit 1
  fi
done

if [ -z "${COMMIT}" ]; then
  COMMIT="$(jq -r '.targetVscodeCommit // empty' "${LOCKFILE}")"
fi

if [ -z "${COMMIT}" ] || [ "${COMMIT}" = "null" ]; then
  echo "ERROR: could not determine VS Code commit." >&2
  exit 1
fi

if ! getent passwd "${REMOTE_USER}" >/dev/null 2>&1; then
  echo "ERROR: remote user not found: ${REMOTE_USER}" >&2
  exit 1
fi

REMOTE_HOME="$(getent passwd "${REMOTE_USER}" | cut -d: -f6)"

if [ -z "${EXTENSIONS_DIR}" ]; then
  EXTENSIONS_DIR="${REMOTE_HOME}/.vscode-server/extensions"
fi

if [ -z "${CODE_SERVER}" ]; then
  CANDIDATE_1="${REMOTE_HOME}/.vscode-server/cli/servers/Stable-${COMMIT}/server/bin/code-server"
  CANDIDATE_2="${REMOTE_HOME}/.vscode-server/bin/${COMMIT}/bin/code-server"

  if [ -x "${CANDIDATE_1}" ]; then
    CODE_SERVER="${CANDIDATE_1}"
  elif [ -x "${CANDIDATE_2}" ]; then
    CODE_SERVER="${CANDIDATE_2}"
  elif command -v code >/dev/null 2>&1; then
    CODE_SERVER="$(command -v code)"
  else
    echo "ERROR: could not find code-server or code CLI." >&2
    echo "Expected one of:" >&2
    echo "  ${CANDIDATE_1}" >&2
    echo "  ${CANDIDATE_2}" >&2
    exit 1
  fi
fi

mkdir -p "${EXTENSIONS_DIR}"
chown -R "${REMOTE_USER}:${REMOTE_USER}" "${EXTENSIONS_DIR}" || true

run_as_user() {
  if command -v runuser >/dev/null 2>&1; then
    runuser -u "${REMOTE_USER}" -- "$@"
  else
    su -s /bin/bash "${REMOTE_USER}" -c "$(printf '%q ' "$@")"
  fi
}

LOCK_DIR="$(cd "$(dirname "${LOCKFILE}")" && pwd)"
REPO_ROOT_GUESS="$(cd "${LOCK_DIR}/../.." 2>/dev/null && pwd || true)"

mapfile -t EXT_IDS < <(jq -r '.containerInstallOrder[]' "${LOCKFILE}")

if [ "${#EXT_IDS[@]}" -eq 0 ]; then
  echo "No container extensions to install."
  exit 0
fi

for EXT_ID in "${EXT_IDS[@]}"; do
  VSIX_REL="$(jq -r --arg id "${EXT_ID}" '.extensions[$id].vsixPath' "${LOCKFILE}")"
  EXPECTED_SHA="$(jq -r --arg id "${EXT_ID}" '.extensions[$id].sha256' "${LOCKFILE}")"
  VERSION="$(jq -r --arg id "${EXT_ID}" '.extensions[$id].version' "${LOCKFILE}")"

  if [ -z "${VSIX_REL}" ] || [ "${VSIX_REL}" = "null" ]; then
    echo "ERROR: missing vsixPath for ${EXT_ID}" >&2
    exit 1
  fi

  # Support lockfiles produced with repo-relative artifact paths and copied to /opt.
  VSIX_PATH="${VSIX_REL}"
  if [ ! -f "${VSIX_PATH}" ]; then
    BASENAME_PATH="$(basename "${VSIX_REL}")"
    FOUND="$(find /opt /workspace /workspaces "${LOCK_DIR}" -type f -name "${BASENAME_PATH}" 2>/dev/null | head -1 || true)"
    if [ -n "${FOUND}" ]; then
      VSIX_PATH="${FOUND}"
    fi
  fi

  if [ ! -f "${VSIX_PATH}" ]; then
    echo "ERROR: VSIX not found for ${EXT_ID}: ${VSIX_REL}" >&2
    exit 1
  fi

  ACTUAL_SHA="$(sha256sum "${VSIX_PATH}" | awk '{print $1}')"

  if [ "${ACTUAL_SHA}" != "${EXPECTED_SHA}" ]; then
    echo "ERROR: SHA256 mismatch for ${EXT_ID}" >&2
    echo "Expected: ${EXPECTED_SHA}" >&2
    echo "Actual:   ${ACTUAL_SHA}" >&2
    exit 1
  fi

  echo "Installing ${EXT_ID}@${VERSION}"
  echo "  ${VSIX_PATH}"

  run_as_user \
    "${CODE_SERVER}" \
      --extensions-dir "${EXTENSIONS_DIR}" \
      --user-data-dir /tmp/vscode-server-user-data \
      --install-extension "${VSIX_PATH}" \
      --force
done

echo "Installed extensions:"
run_as_user \
  "${CODE_SERVER}" \
    --extensions-dir "${EXTENSIONS_DIR}" \
    --user-data-dir /tmp/vscode-server-user-data \
    --list-extensions \
    --show-versions \
  | tee /tmp/vscode-installed-extensions.txt

for EXT_ID in "${EXT_IDS[@]}"; do
  if ! grep -qi "^${EXT_ID}@" /tmp/vscode-installed-extensions.txt; then
    echo "ERROR: extension not found after install: ${EXT_ID}" >&2
    exit 1
  fi
done

chown -R "${REMOTE_USER}:${REMOTE_USER}" "${REMOTE_HOME}/.vscode-server" || true

echo "VS Code extension install complete."
```

Make executable:

```bash
chmod +x scripts/install-vscode-extensions.sh
```

```dockerfile
      --commit "${VSCODE_COMMIT}"

USER vscode

WORKDIR /workspaces
```

# Devcontainer Template Behavior

Do not rely on this in airgap mode:

```jsonc
{
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python"
      ]
    }
  }
}
```

That tells VS Code to install extensions when the container starts. In an airgap environment, that may try to reach the Marketplace.

For offline images, bake extensions into the image instead.

It is acceptable to keep a separate recommendations file:

```text
.vscode/extensions.json
```

but do not rely on it for installation.

# Security and License Notes

The script must preserve metadata for review:

```text
publisher
extension ID
version
download URL
SHA256
license from package.json if present
repository from package.json if present
extensionKind
engines.vscode
dependencies
```

Do not silently install unknown extensions.

Do not silently ignore SHA256 mismatches.

Do not assume that a VSIX install means the extension is fully offline at runtime.

Some extensions may download language servers, analyzers, debug adapters, Java tools, Python tools, or other assets at first activation. Those runtime dependencies must be handled separately if true airgap operation is required.

Examples likely to need additional runtime/tool consideration:

```text
SonarSource.sonarlint-vscode
rust-lang.rust-analyzer
ms-python.python
ms-toolsai.jupyter
vscjava.vscode-java-pack
ms-vscode.cpptools
```

This task only covers the VS Code extension VSIX layer.

---

# Special Handling for Extension Packs

The resolver must expand extension packs.

For example:

```text
vscjava.vscode-java-pack
```

is an extension pack. It should cause the resolver to download the pack itself plus its pack members.

Install order:

```text
pack member dependencies
pack members
extension pack itself
```

Similarly:

```text
ms-vscode-remote.vscode-remote-extensionpack
```

is an extension pack. If its pack members are UI-only or host-side, do not force them into the container. Record them as host-only in the lockfile.
